Domina Git y GitHub

Conceptos

¿Qué es Git?

Git es un sistema de control de versiones distribuido creado por Linus Torvalds en 2005.

📦

Control de Versiones

Registra cambios en tus archivos a lo largo del tiempo, permitiéndote volver a versiones anteriores.

🌐

Distribuido

Cada desarrollador tiene una copia completa del repositorio, permitiendo trabajo sin conexión.

🤝

Colaboración

Múltiples personas pueden trabajar en el mismo proyecto sin conflictos.

🚀

Ramas (Branches)

Crea líneas de desarrollo independientes para experimentar sin afectar el código principal.

Plataforma

¿Qué es GitHub?

GitHub es una plataforma de hosting para repositorios Git que ofrece herramientas de colaboración, gestión de proyectos y código abierto.

Repositorios Remotos

Almacena tu código en la nube y accede desde cualquier lugar.

Pull Requests

Propón cambios y revisa código de tu equipo antes de fusionar.

Issues y Projects

Gestiona tareas, errores y organiza tu trabajo con tableros Kanban.

GitHub Actions

Automatiza flujos de trabajo CI/CD directamente en tu repositorio.

Workflow

Metodología de Trabajo

Dos flujos de trabajo principales para gestionar el desarrollo de software de manera efectiva.

Git Flow

Ideal para proyectos con ciclos de release planificados y equipos grandes.

main
develop
feature/*
release/*
hotfix/*
main Solo código listo para producción
develop Rama de integración, código en desarrollo
feature/* Nueva funcionalidad (desde develop)
release/* Preparación de release (desde develop)
hotfix/* Correcciones urgentes (desde main)

GitHub Flow

Flujo simple y efectivo para equipos que deployan frecuentemente (deploy continuo).

1

Crear Rama

Crea una rama desde main con nombre descriptivo

git checkout -b feature/nueva-funcionalidad
2

Trabajar y Commits

Haz cambios y commts frecuentes con mensajes claros

git add . && git commit -m "Agregar nueva funcionalidad"
3

Abrir Pull Request

Publica tu rama y crea un PR para revisión

4

Revisión y Discusión

El equipo revisa, comenta y solicita cambios si es necesario

5

Merge a Main

Fusiona la rama a main después de aprobada

git checkout main && git merge feature/nueva-funcionalidad
Básico

Comandos Básicos

Los comandos esenciales que todo desarrollador debe conocer para trabajar con Git.

Configuración Inicial

git config --global user.name "Tu Nombre"

Configura tu nombre de usuario global

git config --global user.email "tu@email.com"

Configura tu email global

git config --global init.defaultBranch main

Establece 'main' como rama por defecto

Iniciar Repositorio

git init

Inicializa un nuevo repositorio Git

git clone <url>

Clona un repositorio remoto

Gestión de Cambios

git status

Muestra el estado del repositorio

git add <archivo>

Añade archivos al staging

git add .

Añade todos los cambios al staging

git commit -m "mensaje"

Guarda los cambios con un mensaje

git commit -am "mensaje"

Añade y commitea archivos rastreados

Historial y Ramas

git log

Muestra el historial de commits

git log --oneline

Historial en una línea por commit

git branch

Lista las ramas existentes

git branch -M main

git checkout -b <rama>

Crea y cambia a una nueva rama

Sincronización

git remote -v

Muestra los repositorios remotos

git push -u origin main

Sube la rama main al remoto

git pull

Descarga y fusiona cambios remotos

git fetch

Descarga cambios sin fusionar

git reset --soft HEAD~1

Deshace el último commit (conserva cambios)

git stash

Guarda cambios temporalmente

git merge <rama>

Best Practices

Tips y Recomendaciones

📝

Commits Atómicos

Haz commits pequeños que representen un solo cambio. Es más fácil revertir y entender el historial.

✔ Hacer: Un cambio por commit ✘ Evitar: Múltiples cambios juntos
💬

Mensajes Claros

Escribe mensajes descriptivos que expliquen el "qué" y "por qué" del cambio.

✘ Mal: fix bug
✔ Bien: fix: Corregir error de validación en formulario de registro
🔒

Nunca hacer Force Push

Evita git push --force en repositorios compartidos. Puedes sobrescribir el trabajo de otros.

✔ Usar: git pull --rebase ✘ Evitar: git push --force
🎯

Usar .gitignore

Excluye archivos innecesarios (node_modules, builds, secretos) del repositorio.

✔ Incluir: dependencias, binarios, configs ✘ Evitar: archivos con passwords, keys
📈

Revisar antes de Push

Siempre revisa qué vas a subir usando git status y git diff.

git status && git diff --staged
🛠

Aliases Útiles

Crea alias para comandos frecuentes que ahorran tiempo.

git config --global alias.s "status"
git config --global alias.lg "log --oneline --graph --all"
git config --global alias.co "checkout"
Seguridad

Métodos de Autenticación

GitHub ya no acepta contraseñas de cuenta para operaciones Git. Usa estos métodos recomendados.

🔑

SSH (Secure Shell)

El método más seguro y recomendado. Usa un par de claves pública/privada.

Configuración:

1
Generar clave SSH ssh-keygen -t ed25519 -C "tu@email.com"
2
Copiar clave pública cat ~/.ssh/id_ed25519.pub Copia el contenido mostrado
3
Agregar en GitHub

Settings → SSH and GPG keys → New SSH key

4
Configurar remoto SSH git remote set-url origin git@github.com:usuario/repo.git
✔ Ventajas:
  • Sin necesidad de ingresar credenciales
  • Muy seguro
  • Una vez configurado, funciona siempre
🔑

Token de Acceso Personal (Classic)

Método alternativo usando tokens generados desde GitHub.

Generar Token:

1
Ir a Settings

Settings → Developer settings → Personal access tokens → Tokens (classic)

2
Crear nuevo token

Generate new token (classic)

3
Configurar permisos

Selecciona "repo" para acceso completo

4
Usar el token

Cuando Git pida credentials, usa el token como contraseña

✔ Ventajas:
  • Fácil de generar
  • Puedes configurar permisos específicos
  • Puedes revocar tokens anytime
✘ Desventajas:
  • Debes copiar y pegar el token
  • Expira eventualmente
🔑

GitHub CLI

La forma más moderna usando la CLI de GitHub.

Autenticación con gh:

1
Instalar GitHub CLI brew install gh (macOS)
winget install GitHub.cli (Windows)
2
Iniciar sesión gh auth login
3
Seleccionar opciones

Follow prompts: GitHub.com → SSH → Paste SSH key

✔ Ventajas:
  • Autenticaciónweb interactiva
  • Maneja múltiples cuentas
  • Comandos directamente desde terminal
⚠ Importante: GitHub eliminó la autenticación con contraseña de cuenta desde 2021. Debes usar uno de los métodos acima.
Archivo Especial

Archivo .gitignore

El archivo .gitignore especifica qué archivos y carpetas deben ser ignorados por Git.

¿Qué es?

Un archivo de texto que le dice a Git qué archivos o carpetas no debe rastrear. Esencial para:

  • Evitar subir archivos sensibles (passwords, API keys)
  • Ignorar dependencias instaladas (node_modules, vendor)
  • Excluir archivos decompilación (.class, .pyc, .exe)
  • Mantener limpio el repositorio

  • Usa templates: GitHub ofrece .gitignore templates por lenguaje

  • Ya tracked?: Si un archivo ya está rastreado, agrégalo a .gitignore y luego: git rm --cached archivo

CI/CD con GitHub Actions

GitHub Actions te permite automatizar flujos de trabajo directamente en GitHub.

Ejemplo: Pipeline de CI

name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test
Experto

Comandos Avanzados

Técnicas y comandos avanzados para управления tu historial y flujo de trabajo.

Rebase

Reescribe el historial de commits aplicando los cambios sobre otra rama.

# Rebase de feature sobre main git checkout feature/nueva-func git rebase main
⚠ Nunca hacer rebase en ramas públicas

Cherry-pick

Aplica commits específicos de una rama a otra.

# Aplicar un commit específico git cherry-pick abc1234 # Aplicar múltiples commits git cherry-pick abc1234 def5678

Reflog

Historial de todas las referencias (útil para recuperar cambios "perdidos").

# Ver historial de referencias git reflog # Recuperar estado anterior git checkout HEAD@{2}

Tags (Etiquetas)

Marcadores de versiones específicas en el historial.

# Tag lightweight git tag v1.0.0 # Tag anotado (recomendado) git tag -a v1.0.0 -m "Versión 1.0.0" # Subir tags al remoto git push origin v1.0.0

Git Hooks

Scripts que se ejecutan en momentos específicos del flujo de Git.

# Ubicación de hooks .git/hooks/ # Hook pre-commit ejemplo #!/bin/bash npm test
💡 Usa husky para gestionar hooks en proyectos

Worktree

Trabaja en múltiples ramas simultáneamente en diferentes carpetas.

# Crear nuevo worktree git worktree add ../rama-nueva main # Listar worktrees git worktree list # Eliminar worktree git worktree remove ../rama-nueva

Cómo contribuir a proyectos externos y manejar repositorios forked.

Crea Rama para Cambios

Haz Cambios y Commit

🔄 Sincronizar Fork con Upstream

git fetch upstream Descargar cambios del repo original
git checkout main Ir a tu rama main
git merge upstream/main Fusionar cambios
git push origin main Subir a tu fork
Troubleshooting

Errores Comunes y Soluciones

⚠ "detached HEAD"

Estás en un commit específico, no en una rama.

Solución: git checkout main # Volver a main git checkout -b nueva-rama # Crear rama
⚠ "Merge Conflict"

Git no puede fusionar automáticamente los cambios.

Solución: # 1. Abre los archivos en conflicto # 2. Resuelve los conflictos manualmente # 3. git add archivo-resuelto # 4. git commit
⚠ "refusing to merge unrelated histories"

El repositorio local y remoto tienen historiales diferentes.

Solución: git pull origin main --allow-unrelated-histories
⚠ "push rejected"

Tu rama está detrás de la remota.

Solución: git pull --rebase origin main git push origin main
⚠ "fatal: bad object"

Archivo .git corrupto o descargado incorrectamente.

Solución: git fsck --full # Si no funciona, reclonar el repo
⚠ "error: pathspec did not match"

La rama o archivo especificado no existe.

Solución: git branch -a # Ver ramas disponibles git fetch --all # Actualizar refs

Glosario de Términos

Repository

Directorio que contiene el proyecto y su historial de versiones.

Commit

Snapshot de los archivos en un momento específico.

Branch

Línea de desarrollo independiente.

Merge

Fusionar cambios de una rama a otra.

Rebase

Reescribir historial aplicando commits sobre otra base.

HEAD

Puntero al commit/rama actual.

Origin

Nombre por defecto del repositorio remoto.

Upstream

Repositorio original del que hiciste fork.

Staging Area

Área donde se preparan los archivos para el commit.

Working Directory

Tu carpeta de trabajo con archivos actuales.

Pull Request

Solicitud para fusionar cambios a otro repositorio.

Fork

Copia independiente de un repositorio.

Clone

Copiar un repositorio remoto a local.

Stash

Guardar cambios temporalmente sin commit.

Tag

Marcador de versión específica.

Remote

Servidor donde se guarda el repositorio.
